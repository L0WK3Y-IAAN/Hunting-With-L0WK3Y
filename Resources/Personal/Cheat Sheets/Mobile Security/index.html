Overview: Mobile Application Penetration Testing

What is Mobile Application Penetration Testing?

Mobile application penetration testing is a comprehensive security assessment of iOS and Android applications designed to identify vulnerabilities, misconfigurations, and security weaknesses before attackers exploit them. It simulates real-world attacks to evaluate:

  • Data storage and encryption mechanisms
  • Authentication and session management
  • Communication security (TLS/encryption)
  • Authorization and access controls
  • Platform-specific security controls (Keychain, Keystore, etc.)
  • Code integrity and reverse engineering resistance
  • Backend API security and data validation
  • Third-party library vulnerabilities and supply chain risks

Testing Scope Considerations

  • App Type: Native Android (Java/Kotlin), Native iOS (Swift/Obj-C), or Hybrid/Cross-Platform (React Native, Flutter)
  • OS Versions: Supported OS versions and version-specific vulnerabilities
  • Device Types: Phones, tablets, wearables, or other form factors
  • Backend Services: APIs, authentication services, data storage
  • Third-Party Integrations: SDKs, libraries, and external services
  • Data Sensitivity: Type and classification of data processed
  • Regulatory Requirements: HIPAA, PCI-DSS, GDPR, SOC 2, etc.

Testing Methodology & Framework

OWASP MASTG Methodology Overview

The standard MASTG approach includes these phases:

  • 1. Intelligence Gathering & Environmental Analysis
    Understanding the app architecture, environmental context, OS versions, backend services, and data sensitivity.
  • 2. Mapping & Architecture Review
    Automated scanning combined with manual exploration to identify entry points, data flows, key components, and potential vulnerabilities.
  • 3. Threat Modeling & Risk Prioritization
    Creating threat models (using STRIDE or similar), identifying high-risk areas, and prioritizing tests based on impact/likelihood.
  • 4. Test Case Development
    Creating specific, reproducible test cases aligned with MASVS controls and potential attack vectors.
  • 5. Static Analysis
    Decompiling, reverse engineering, and reviewing source code and bytecode for vulnerabilities.
  • 6. Dynamic Analysis
    Runtime inspection, traffic interception, API testing, and behavioral analysis of the app in action.
  • 7. Exploitation & PoC Development
    Attempting to exploit identified vulnerabilities and creating proofs-of-concept to validate findings.
  • 8. Reporting & Remediation Guidance
    Documenting findings with severity, impact, remediation steps, and retesting to verify fixes.

    Rooting & Jailbreaking Considerations

    ‚ö†Ô∏è Legal & Ethical Boundaries

    Always obtain explicit written authorization before rooting/jailbreaking devices. This is especially important when:

    • Testing third-party or production apps
    • Devices are owned by organizations
    • Warranty implications exist

    Android Rooting

    • Magisk: Systemless root with MagiskHide to bypass root detection
    • SuperSU: Traditional rooting method (older approach)
    • ADB Root: On development builds (adb root)
    • Emulator Root: Android Studio emulator with -writable-system flag

    iOS Jailbreaking

    • Checkra1n: Most reliable for lower iOS versions (A7-A11 devices)
    • Palera1n: Newer alternative for recent iOS versions
    • Sileo/Cydia: Package managers for jailbroken devices
    • Runtime Modification: Frida + Objection without full jailbreak

    Proxying & Interception Setup

    HTTP(S) Interception Architecture

    • Install proxy CA certificate in device trust store
    • Configure device WiFi proxy settings (or use VPN-based proxies)
    • Enable Burp Suite/ZAP listener on attack machine
    • Route traffic through proxy: device ‚Üê proxy ‚Üê backend
    • Inspect, log, modify, and manipulate requests/responses

    üìå Certificate Pinning Challenge

    Apps may implement certificate pinning to prevent MITM attacks. Bypass techniques include:

    • Frida hooking to disable pinning logic
    • Modifying APK/IPA to disable checks (static bypass)
    • Using tools like Burp Suite's pinning bypass plugins
    • Network-level manipulation if pinning is weak

    Decompiling & Reverse Engineering Workflow

    Android APK Analysis Workflow

    # 1. Obtain APK
adb pull /data/app/com.example.app/base.apk ./target.apk

# 2. Extract resources & smali (bytecode)
apktool d target.apk

# 3. Decompile to readable Java
jadx target.apk
jadx -gui target.apk

# 4. Search for sensitive strings (keys, URLs, etc.)
grep -r "secret\|password\|token\|key" ./target/

# 5. Analyze AndroidManifest.xml
- Check declared activities, services, broadcast receivers, content providers
- Review permissions requested
- Identify exported components

# 6. Review classes of interest:
- Authentication/login activities
- Data storage handlers
- Network communication
- Cryptography implementations

# 7. Use Frida for runtime verification
frida -U -f com.example.app --no-pause -l hook.js

    iOS IPA Analysis Workflow

    # 1. Decrypt App Store IPA (if needed)
clutch -i (find app GUID)
clutch -d GUID

# 2. Extract IPA contents
unzip target.ipa

# 3. Locate main executable
cd Payload/App.app/

# 4. Disassemble with Ghidra/Radare2
ghidra ./App_executable

# 5. Analyze with Radare2
r2 ./App_executable
aaa  (analyze all)
afl  (list functions)
pdf @ main (print disassembly at main)

# 6. String analysis
strings ./App_executable | grep -i "secret\|api\|token"

# 7. Examine Info.plist
plutil -p Info.plist  (view configuration)

    Common Static Analysis Findings

    Hardcoded Secrets

    • API keys, tokens, or credentials in source code
    • Database connection strings
    • Encryption keys or passwords
    • Impact: Complete credential compromise; unauthorized API access

    Insecure Data Storage

    • Sensitive data in SharedPreferences (Android) without encryption
    • Plaintext data in app cache or temp files
    • Unencrypted UserDefaults (iOS)
    • Database files without encryption
    • Impact: Data theft from stolen device or forensic analysis

    Weak Cryptography

    • Use of deprecated algorithms (DES, MD5, SHA1)
    • Custom encryption implementations instead of standard libraries
    • Hardcoded IV or encryption keys
    • Insufficient key derivation (weak PBKDF2 iteration counts)
    • Impact: Data decryption; authentication bypass

    Insecure Deserialization

    • Deserializing untrusted data without validation
    • Using unsafe Java serialization
    • Impact: Remote code execution; arbitrary object instantiation

    Exported Components (Android)

    • Activities, services, or broadcast receivers exposed without intent filters
    • Content providers without proper permissions
    • Impact: Unauthorized component invocation; data leakage

    Vulnerable Dependencies

    • Outdated libraries with known CVEs
    • Transitive dependencies not updated
    • Impact: Known vulnerability exploitation from third-party code

    Common Dynamic Testing Workflows

    Workflow: API Endpoint Testing

    1. Install Burp Certificate Authority in device trust store
    2. Configure WiFi proxy to point to Burp listener
    3. Start app and navigate through user flows
    4. Capture all API requests/responses in Burp
    5. Analyze request patterns:
      • Authentication tokens & session management
      • User ID enumeration (sequential IDs, GUIDs)
      • Request signing & integrity checks
      • Rate limiting & brute-force protections
    6. Test for vulnerabilities:
      • Modify request parameters (user_id, page numbers)
      • Test missing authorization checks
      • Attempt privilege escalation
      • Bypass input validation
    7. Document findings with reproducible steps & PoC

    Workflow: Runtime Hook (Frida Example)

    1. Identify target method (e.g., password validation)
    2. Write Frida hook script (JavaScript)
    3. Load hook into running process: frida -U -f com.app.package --no-pause -l hook.js
    4. Hook modifies method behavior (bypass check, log credentials, etc.)
    5. Observe modified behavior & document bypass

    Backend API & Service Testing

    Mobile API Architecture & Testing Focus

    Mobile apps communicate with backend APIs for data retrieval, authentication, business logic, and data persistence. Testing focuses on:

    Authentication & Session Management

    Common Vulnerabilities

    • Weak Token Generation: Predictable or short-lived tokens
    • Missing Token Refresh: No mechanism to refresh expired tokens
    • Token Leakage: Tokens in logs, error messages, or cache
    • No Token Binding: Token valid for any device/IP
    • Insecure Token Storage: No encryption in transit or at rest
    • Session Fixation: Token reuse without invalidation
    • Missing Logout: Tokens not invalidated on logout

    Testing Procedure

    1. Capture authentication request (login) via proxy
    2. Analyze response for token, session ID, or secret
    3. Test token validity period & refresh mechanism
    4. Attempt token reuse after logout
    5. Modify token claims (if JWT):
      • Change user_id in payload
      • Extend expiration time
      • Remove signature verification (test server-side validation)
    6. Test token leakage in error responses, logs, or caches
    7. Verify token binding to device/user/IP

    Authorization & Access Control

    Common Vulnerabilities

    • Insecure Direct Object Reference (IDOR): Modify user_id to access other user data
    • Broken Function-Level Authorization: Admin functions callable by regular users
    • Privilege Escalation: Upgrade user role via API manipulation
    • Missing Authorization Checks: No verification of user rights before data access
    • Parameter Pollution: Multiple parameters for same functionality with inconsistent checks

    Testing Procedure

    1. Identify all API endpoints and their functionality
    2. Map endpoints to user roles & permissions
    3. For each endpoint, test as different users/roles
    4. Attempt to access data/functions outside your scope:
      • Change user_id parameters to other users
      • Modify role/permission fields in requests
      • Try accessing admin endpoints
    5. Document all unauthorized access succeeds

    Input Validation & Injection Attacks

    Common Vulnerabilities

    • SQL Injection: Manipulate query logic via app-supplied data
    • NoSQL Injection: JSON-based injection for database queries
    • XML External Entity (XXE): XML parsing exploits for data exfiltration
    • Command Injection: OS command execution via unsanitized inputs
    • Path Traversal: Access files outside intended directory
    • API Logic Bypasses: Logic flaws in business logic validation

    Testing Procedure

    1. Identify input points (user ID, search terms, filters, file uploads)
    2. Test basic injection payloads:
      • SQL: ' OR '1'='1
      • NoSQL: {"$ne": null}
      • XXE: Entity declarations for file reads
    3. Use payload lists (SecLists, Burp intruder)
    4. Observe response time, error messages, and behavior changes
    5. Escalate findings to command execution, data exfiltration

    Mobile Security Vulnerabilities & Weakness Patterns

    OWASP Mobile Top 10 2024 & Detailed Mappings

    1. Improper Credential Usage

    • Hardcoded Credentials: API keys, tokens, or secrets embedded in code
    • Weak Credential Storage: Plaintext passwords in SharedPreferences or UserDefaults
    • Credential Leakage: Credentials in logs, error messages, or backups
    • Testing: Static analysis for hardcoded values; check logs/backups for leakage
    • MASVS Control: MASVS-STORAGE-2 (Cryptographic Practices)

    2. Inadequate Supply Chain Security

    • Vulnerable Dependencies: Outdated libraries with known CVEs
    • Malicious Libraries: Compromised third-party SDKs or packages
    • Transitive Dependencies: Vulnerabilities in indirect dependencies
    • Testing: SCA tools (Snyk, Dependabot); manual dependency audit
    • MASVS Control: MASVS-CODE-1 (Weakly Controlled Third-Party Libraries)

    3. Inadequate Authentication/Authorization

    • Weak Authentication: Insufficient protection for login credentials
    • Broken Authorization: Users access data/functions outside their scope
    • Session Management Flaws: Token reuse, no expiration, session fixation
    • Privilege Escalation: Regular users gain admin access
    • Testing: API testing for IDOR, authorization checks, session tampering
    • MASVS Control: MASVS-AUTH (Authentication & Session Management)

    4. Insecure Communication

    • Unencrypted Data Transmission: HTTP instead of HTTPS
    • Weak TLS Configuration: Outdated cipher suites, no certificate validation
    • Certificate Pinning Bypass: App allows MITM by not pinning certificates
    • Man-in-the-Middle (MITM) Risk: Interception of sensitive data in transit
    • Testing: Proxy interception, TLS analysis, certificate validation bypass
    • MASVS Control: MASVS-NETWORK (Network Communication Security)

    5. Inadequate Cryptography

    • Weak Algorithms: DES, RC4, MD5, SHA1, or custom encryption
    • Insufficient Key Length: Keys too short for algorithm requirements
    • Hardcoded Keys: Encryption keys embedded in code
    • Weak Key Derivation: Insufficient PBKDF2 iterations, no salt
    • Insecure Random: java.util.Random instead of SecureRandom
    • Testing: Static analysis for algorithm review; cryptographic testing
    • MASVS Control: MASVS-CRYPTO (Cryptographic Practices)

    6. Insecure Data Storage

    • Plaintext Storage: Sensitive data stored unencrypted
    • SharedPreferences/UserDefaults: No encryption for sensitive keys
    • Database Files: SQLite databases without encryption
    • Cache & Temp Files: Sensitive data left in app cache or temp storage
    • Backups: Sensitive data backed up to cloud or accessible backups
    • Testing: File system analysis; encrypted storage verification
    • MASVS Control: MASVS-STORAGE (Secure Data Storage)

    7. Insufficient Logging & Monitoring

    • Lack of Audit Logs: No logging of security-relevant events
    • Sensitive Data in Logs: Credentials or PII exposed in application logs
    • No Alerting: Suspicious activities not monitored or alerted
    • Testing: Logcat review for sensitive data; security event verification
    • MASVS Control: MASVS-PRIVACY & MASVS-CODE

    8. Insecure Code Quality

    • Memory Issues: Buffer overflows, use-after-free (native code)
    • Deserialization Flaws: Insecure deserialization leading to RCE
    • SQL Injection: Unsanitized SQL queries
    • Command Injection: OS command execution via unsanitized input
    • Path Traversal: Access files outside intended directory
    • Testing: SAST tools, input fuzzing, code review
    • MASVS Control: MASVS-CODE (Code Quality)

    9. Weak Reverse Engineering Protections

    • No Obfuscation: Code easily readable after decompilation
    • Missing Root/Jailbreak Detection: App doesn't detect compromised devices
    • Debuggable Apps: Debugflag set in manifest (Android)
    • No Integrity Checks: App code not verified at runtime
    • Testing: APK/IPA analysis; dynamic tampering attempts; root/jailbreak detection
    • MASVS Control: MASVS-RESILIENCE (Resilience to Reverse Engineering)

    10. Misuse of Platform Features

    • Insecure Custom URL Schemes: No validation of deep link parameters (iOS)
    • Exported Content Providers: Accessible to other apps without permission (Android)
    • WebView Vulnerabilities: Insecure JavaScript bridge, no input validation
    • Biometric Bypass: Weak fallback authentication after biometric failure
    • Keychain/Keystore Misuse: Sensitive data not properly secured
    • Testing: Component interaction testing, WebView analysis, platform-specific checks
    • MASVS Control: MASVS-PLATFORM (Platform-Specific Security Controls)

    SAST (Static Application Security Testing) for Mobile

    SAST in Mobile CI/CD Pipeline

    • When: On every commit/pull request
    • What to Scan:
      • Java/Kotlin source (Android)
      • Swift/Objective-C source (iOS)
      • JavaScript/Dart (Hybrid apps)
    • Tools:
      • SonarQube (comprehensive code quality + security)
      • Checkmarx (enterprise SAST)
      • Snyk (developer-friendly)
      • Semgrep (fast, customizable rules)
    • Gate Criteria:
      • Block on Critical/High vulnerabilities
      • Warn on Medium severity
      • Allow Low findings with documentation

    Example: GitHub Actions SAST Pipeline

    name: SAST Security Scan

on: [push, pull_request]

jobs:
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Run SonarQube Scan
        uses: SonarSource/sonarqube-scan-action@master
        env:
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
      
      - name: Run Semgrep
        uses: returntocorp/semgrep-action@v1
        with:
          config: p/security-audit

      - name: Check Quality Gate
        if: failure()
        run: exit 1

    SCA (Software Composition Analysis)

    Dependency Scanning in Mobile Apps

    • Continuously Monitor: All direct and transitive dependencies
    • Tools:
      • Snyk (most comprehensive for mobile)
      • Dependabot (GitHub native, free)
      • WhiteSource/Mend (enterprise SCA)
      • OWASP Dependency-Check (open-source)
    • Scanning Scope:
      • Gradle/Maven dependencies (Android)
      • CocoaPods/SPM (iOS)
      • npm/yarn packages (Hybrid)
    • Action on Finding:
      • Update vulnerable packages automatically (or PR)
      • Block deployment if critical CVE found
      • License compliance checks

    DAST (Dynamic Application Security Testing)

    Dynamic Testing in CI/CD

    • When: After build, on staging environment
    • What to Test:
      • App deployed on emulator/real device
      • API endpoints interception & testing
      • Authentication/authorization validation
      • Runtime behavior analysis
    • Tools:
      • Appknox (mobile-first automated DAST)
      • MobSF (open-source dynamic analysis)
      • Burp Suite with automation (Repeater, Intruder)
      • Frida with custom scripts
    • Typical Duration: 30-90 minutes per full scan

    Example: Automated DAST with MobSF

    # Pipeline stage: Dynamic Analysis
name: Dynamic Security Testing

jobs:
  dast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Start MobSF Server
        run: docker run -d -p 8000:8000 mobsf/mobsf:latest

        run: |

            -H "Authorization: Token $MOBSF_API_KEY"
      
      - name: Generate Report
        run: |
          curl http://localhost:8000/api/v1/report \
            -H "Authorization: Token $MOBSF_API_KEY" > report.json
      
      - name: Parse & Gate
        run: |
          # Parse report, fail if critical findings
          python3 check_security_gates.py report.json

    Complete CI/CD Pipeline Example (GitHub Actions)

    name: Mobile AppSecOps Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]
  schedule:
    - cron: '0 2 * * 0'  # Weekly scan

jobs:
  sast:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: SAST Scan
        run: |
          # SonarQube scan
          sonar-scanner \
            -Dsonar.projectKey=mobile-app \
            -Dsonar.sources=. \
            -Dsonar.host.url=$SONAR_HOST_URL \
            -Dsonar.login=$SONAR_TOKEN
      - name: Fail on High Severity
        run: |
          # Check quality gate result
          curl -u $SONAR_LOGIN: \
            $SONAR_HOST_URL/api/qualitygates/project_status?projectKey=mobile-app \
            | jq -e '.projectStatus.status == "OK"'

  sca:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Snyk Scan
        env:
          SNYK_TOKEN: ${{ secrets.SNYK_TOKEN }}
        run: |
          npm install -g snyk
          snyk test --json > snyk-report.json || true
          snyk-severity-check snyk-report.json || exit 1

  build:
    runs-on: ubuntu-latest
    needs: [sast, sca]
    steps:
      - uses: actions/checkout@v3
      - name: Build Android
        run: ./gradlew assembleRelease
      - name: Verify Signing
        run: |
          jarsigner -verify -verbose build/outputs/apk/release/*.apk
      - name: Upload Artifact
        uses: actions/upload-artifact@v3
        with:
          name: release-apk
          path: build/outputs/apk/release/*.apk

  dast:
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/download-artifact@v3
        with:
          name: release-apk
      - name: Start Emulator
        run: |
          # Start Android emulator
          emulator -avd test-device -no-window &
      - name: Run MobSF DAST
        run: |
          docker run -p 8000:8000 mobsf/mobsf &
          sleep 30
          python3 run_dast_tests.py
      - name: Upload DAST Report
        if: always()
        uses: actions/upload-artifact@v3
        with:
          name: dast-report
          path: dast-results/

  security-gate:
    runs-on: ubuntu-latest
    needs: [sast, sca, dast]
    if: always()
    steps:
      - name: Aggregate Results
        run: |
          # Combine SAST, SCA, DAST findings
          python3 aggregate_findings.py
      - name: Generate Report
        run: |
          python3 generate_security_report.py
      - name: Notify Slack
        if: failure()
        uses: slackapi/slack-github-action@v1
        with:
          webhook-url: ${{ secrets.SLACK_WEBHOOK }}
          payload: |
            {
              "text": "Security findings detected in #${{ github.run_number }}"
            }

    Vulnerability Classification & Severity Scoring

    Severity CVSS Score Characteristics Remediation Timeline
    Critical 9.0-10.0 Immediate exploitation without authentication; direct data breach/RCE Immediate (1-2 days)
    High 7.0-8.9 Requires authentication or additional steps; significant impact Urgent (1-2 weeks)
    Medium 4.0-6.9 Limited scope; requires specific conditions; moderate impact Standard (1-2 months)

    Remediation Guidance Examples

    Example 1: Hardcoded API Key

    Vulnerability: API key hardcoded in BuildConfig.java

    Remediation:

    • Move API key to secure backend configuration
    • Implement key rotation mechanism
    • Use environment-specific secrets (staging vs. production)
    • Implement API rate limiting on backend
    • Revoke compromised key immediately

    Example 2: Unencrypted Sensitive Data Storage

    Vulnerability: User tokens stored in SharedPreferences without encryption

    Remediation:

    • Use EncryptedSharedPreferences (AndroidX Security library)
    • Implement Android Keystore for key management
    • Use AES-256-GCM encryption
    • Test encryption on all supported API levels

    Example 3: Missing Authorization Check

    Vulnerability: API endpoint allows accessing other user's data (IDOR)

    Remediation:

    • Server-side authorization check before returning data
    • Verify request user_id matches requested user_id
    • Implement role-based access control (RBAC)
    • Log all data access attempts
    • Implement rate limiting per user
